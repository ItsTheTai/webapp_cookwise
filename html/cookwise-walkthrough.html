1 · Nachschlagewerk

JavaScript-Vokabular für diesen Code

Diese Bausteine tauchen in fast jedem Code-Block wieder auf. Einmal hier nachlesen — die Annotationen in den folgenden Abschnitten verweisen darauf zurück, statt alles jedes Mal neu zu erklären.

const let

Variable, die man nicht (const) bzw. doch (let) später neu zuweisen kann.

x => { … }

Arrow-Funktion — Kurzschreibweise für eine Funktion. response => {…} heißt: „eine Funktion, die einen Parameter namens response entgegennimmt“.

fetch(url)

Schickt eine Anfrage los und liefert sofort ein Promise zurück — ein „Versprechen“ auf eine Antwort, die erst später ankommt.

.then(fn)

„Wenn das Promise fertig ist, führe fn mit dem Ergebnis aus.“ Mehrere .then() lassen sich aneinanderhängen (Kette).

response.ok .status

ok ist true/false, ob die HTTP-Antwort ein Erfolg war (Status 200–299). status ist die genaue Zahl, z. B. 404.

response.json()

Wandelt den Antworttext in ein echtes JavaScript-Objekt um — liefert selbst wieder ein Promise, deshalb meist mit return in einem .then().

`Text ${variable}`

Template-String (Backticks): Text und Variablenwerte kombinieren, ohne +-Verkettung.

.map .filter .find .some .forEach

Eingebaute Array-Methoden: verwandeln, aussieben, ersten Treffer suchen, „gibt's mindestens einen Treffer?“, „für jedes Element etwas tun“.

?. Optional Chaining

„Greif nur zu, wenn links davon nicht null/undefined ist“ — sonst undefined statt Programmabsturz.

?? Nullish Coalescing

„Nimm den rechten Wert nur, wenn der linke null oder undefined ist.“

el.addEventListener("click", fn)

„Wenn X passiert (Klick, Eingabe, Änderung …), führe fn aus.“

JSON.parse() JSON.stringify()

Text → JS-Objekt bzw. JS-Objekt → Text. Gebraucht, weil localStorage nur Text speichern kann.

a ? b : c Ternärer Operator

Kurzform für „wenn a, dann b, sonst c“ — als Ausdruck, der direkt einen Wert liefert (nicht als ganze if-Anweisung).

localStorage

Speicher im Browser, der Neuladen und sogar Schließen des Browsers übersteht.

3 · Vorher kurz geklärt

Warum überhaupt ein Server?

fetch() auf eine lokale Datei (file://…) blockiert Chrome/Edge per CORS. Deshalb liegt start-server.ps1 bei — ein kleiner Webserver ganz ohne Node/Python, gebaut mit System.Net.HttpListener. Er liefert die Dateien über http://localhost:5500 aus, dort funktioniert fetch() normal.

4 · Die Datenquelle

api.js — die einzige Stelle, die die URL kennt

Weder die Übersicht noch die Detailseite rufen fetch() direkt auf. Beide nutzen zwei Funktionen aus api.js. Dahinter steckt die echte API unter recipes.digitalhumanities.io — mit zwei Eigenheiten, auf die der Code eingeht.

① Paginierung

Die Liste kommt seitenweise (20 Rezepte pro Seite, Feld next zeigt auf die nächste). Für 150 Rezepte braucht es also mehrere Anfragen, nicht eine.

② Liste ≠ Detail

Die Listen-Antwort enthält keine Zutaten/Schritte, nur Kurzinfos. Die gibt es erst über .../rezepte/<id>/ — ein Grund mehr, den fetch-Aufruf zu kapseln.

js/api.js
const API_URL = "https://recipes.digitalhumanities.io/api/rezepte/";
Konstante mit der Basis-Adresse — die einzige Stelle im ganzen Projekt, die diese URL kennt.
// Alle Seiten nacheinander laden und zu einem Array zusammenfügen
function ladeAlleRezepte() {
  return ladeSeite(API_URL, []);
}
Startet die Sammlung: ruft ladeSeite() mit der Basis-URL und einem leeren Array auf — „bisher gesammelt: nichts“.
function ladeSeite(url, bisherGesammelt) {
Zwei Parameter: welche Seite gerade geladen wird (url) und was bisher an Rezepten gesammelt wurde.
  return fetch(url)
    .then(response => {
      if (!response.ok) {
        throw new Error("Server antwortet mit Status " + response.status);
      }
      return response.json();
    })
fetch(url) schickt die Anfrage los. response => {…} läuft, sobald die Antwort da ist (siehe Vokabular). !response.ok prüft auf Server-Fehler und throw „wirft“ dann einen Fehler, der später im .catch() ankommt. response.json() macht aus dem Antworttext ein JS-Objekt.
    .then(seite => {
      const alleBisher = bisherGesammelt.concat(seite.results);
      return seite.next ? ladeSeite(seite.next, alleBisher) : alleBisher;
    });
}
seite ist jetzt {count, next, results, …}. .concat() hängt die neuen Ergebnisse ans bisherige Array. seite.next ? A : B heißt: gibt es eine nächste Seite, ruft sich die Funktion selbst nochmal auf — sonst ist alleBisher das fertige Ergebnis.
// Für ein einzelnes Rezept gezielt den Detail-Endpunkt abrufen (mit Zutaten/Schritten)
function ladeRezeptNachId(id) {
  return fetch(API_URL + id + "/").then(response => {
Für die Detailseite reicht ein gezielter fetch() direkt auf .../rezepte/<id>/ — kein Sammeln nötig.
    if (response.status === 404) {
      return null; // kein Rezept mit dieser ID vorhanden
    }
404 heißt „nicht gefunden“. Statt eines Fehlers wird bewusst null zurückgegeben, damit rezept.js sauber „kein Rezept gefunden“ anzeigen kann, statt abzustürzen.
    if (!response.ok) {
      throw new Error("Server antwortet mit Status " + response.status);
    }
    return response.json();
  });
}
Alle anderen Fehler (Server down, Timeout …) landen weiter im .catch() der aufrufenden Seite.

Beide Funktionen behalten nach außen dieselbe „Signatur“ wie zuvor (ladeAlleRezepte() liefert ein Array, ladeRezeptNachId(id) ein Rezept oder null) — uebersicht.js und rezept.js mussten deshalb kaum angefasst werden.

5 · Übersichtsseite

Aus JSON werden Kacheln

uebersicht.js ruft beim Laden ladeAlleRezepte() auf. Kommt die Antwort an, wird für jedes Rezept eine Bootstrap-Karte gebaut und in #rezept-liste eingehängt.

js/uebersicht.js
ladeAlleRezepte()
  .then(data => {
    alleRezepte = data;
    fuelleFilterOptionen(alleRezepte);
    rezepteAnzeigen(alleRezepte);
  })
ladeAlleRezepte() liefert ein Promise. data => {…} läuft, sobald die Rezepte da sind — data ist dann das fertige Array. Es landet in der globalen Variable alleRezepte (für Filter/Suche später) und wird direkt angezeigt.
  .catch(error => {
    console.error("Fehler beim Laden der Rezepte:", error);
    statusElement.textContent = "Die Rezepte konnten nicht geladen werden. Bitte später erneut versuchen.";
  });
Nur ein .catch() für die ganze Kette: fängt jeden Fehler ab, egal ob er in fetch(), response.json() oder in api.js passiert ist.
// Baut aus einem Rezept-Objekt eine Bootstrap-Card ("Kachel")
function erzeugeKachel(rezept) {
  const spalte = document.createElement("div");
  spalte.className = "col";
document.createElement("div") baut ein neues, noch unsichtbares HTML-Element im Speicher. className gibt ihm eine CSS-Klasse.
  spalte.innerHTML = `
    <a href="rezept.html?id=${rezept.id}" class="text-decoration-none">
      <div class="card recipe-card">
        <img src="${rezept.bild_url ?? "data/" + rezept.bild}" class="card-img-top" alt="${rezept.titel}">

          <p class="card-text text-muted">${rezept.kurzbeschreibung}</p>

  `;
Template-String über mehrere Zeilen (Backticks): ${rezept.id}, ${rezept.titel} … werden durch die echten Werte ersetzt. innerHTML setzt diesen fertigen HTML-Text als Inhalt des Elements.
  return spalte;

Gibt das fertige Karten-Element zurück, das anschließend ins DOM eingehängt wird.

rezept.id landet automatisch im Link (?id=${rezept.id}) — niemand trägt IDs von Hand ein, das erfüllt genau das Kriterium „Navigation Übersicht → Detail funktioniert dynamisch“.

6 · Rendern

Anzeigen — und der leere Fall

rezepteAnzeigen() leert zuerst die Liste, zeigt dann entweder die Kacheln oder eine Meldung, wenn nichts passt.

js/uebersicht.js
function rezepteAnzeigen(rezepte) {
  listeElement.innerHTML = "";
Leert zuerst die bisherige Liste — sonst würden sich alte und neue Kacheln bei jedem Filterdurchlauf vermischen.
  if (rezepte.length === 0) {
    statusElement.textContent = "Keine Rezepte gefunden. Filter anpassen und erneut versuchen.";
    return;
  }
return ohne Wert beendet die Funktion sofort — der Code danach (die Kacheln bauen) wird dann übersprungen.
  statusElement.textContent = "";
  rezepte.forEach(rezept => {
    listeElement.appendChild(erzeugeKachel(rezept));
  });
}
.forEach() ruft die Funktion für jedes Element im Array auf, ohne (anders als .map()) ein neues Array zu erzeugen. appendChild hängt ein Element tatsächlich sichtbar ins DOM.
7 · Interaktion

Filter & Suche — ohne Neuladen

Alle Rezepte liegen bereits im Speicher (alleRezepte). Filter und Suche laufen deshalb komplett im Browser, mit .filter() auf drei Bedingungen gleichzeitig.

js/uebersicht.js
function filternUndAnzeigen() {
  const kategorie = filterKategorie.value;
  const kueche = filterKueche.value;
  const suchbegriff = filterSuche.value.trim().toLowerCase();
Liest die aktuellen Werte der drei Formularfelder aus. .trim() entfernt Leerzeichen am Rand, .toLowerCase() macht Groß-/Kleinschreibung egal.
  const gefiltert = alleRezepte.filter(rezept => {
    const passtKategorie = !kategorie || rezept.kategorie === kategorie;
    const passtKueche = !kueche || rezept.kueche === kueche;
.filter() erzeugt ein neues Array mit nur den Elementen, für die die Funktion true zurückgibt — alleRezepte selbst bleibt unverändert. !kategorie ist true, wenn nichts ausgewählt ist („Alle Kategorien“) — dann passt automatisch alles.
    // rezept.zutaten gibt es in der Übersicht nicht (nur auf der Detailseite) -
    // "?." verhindert einen Absturz und lässt die Titelsuche trotzdem funktionieren
    const passtSuche =
      !suchbegriff ||
      rezept.titel.toLowerCase().includes(suchbegriff) ||
      (rezept.zutaten?.some(zutat => zutat.name.toLowerCase().includes(suchbegriff)) ?? false);

    return passtKategorie && passtKueche && passtSuche;
  });
?. (Optional Chaining) verhindert einen Absturz, wenn zutaten gar nicht existiert. ?? false sorgt dafür, dass am Ende garantiert true/false rauskommt, nie undefined.
  rezepteAnzeigen(gefiltert);
}
Zeigt nur die gefilterte Teilmenge an — die Originaldaten in alleRezepte bleiben für den nächsten Filterdurchlauf erhalten.
filterKategorie.addEventListener("change", filternUndAnzeigen);
filterKueche.addEventListener("change", filternUndAnzeigen);
filterSuche.addEventListener("input", filternUndAnzeigen);
change feuert bei Dropdowns nach der Auswahl, input bei jedem Tastendruck im Suchfeld — deshalb reagiert die Suche sofort beim Tippen, ganz ohne Absenden-Button.

Die Zutatensuche (rezept.zutaten?.some(...)) würde bei der echten API in der Übersicht nie greifen, weil zutaten dort schlicht fehlt — ?. sorgt dafür, dass daraus kein Fehler wird, sondern die Suche einfach nur auf dem Titel weiterläuft. Mit den lokalen Testdaten (die zutaten überall mitliefern) funktioniert die Zutatensuche weiterhin.

8 · Übergang

Übersicht → Detail: die ID in der URL

Der Link auf jeder Kachel zeigt auf rezept.html?id=53276. Die Detailseite liest genau diesen Query-Parameter mit der eingebauten URLSearchParams-API aus — kein manuelles String-Zerschneiden nötig.

js/rezept.js
const params = new URLSearchParams(window.location.search);
const rezeptId = params.get("id");
window.location.search ist der Teil der URL ab dem ?. URLSearchParams macht daraus ein Objekt, aus dem sich einzelne Parameter mit .get("id") auslesen lassen.
if (!rezeptId) {
  statusElement.textContent = "Keine Rezept-ID angegeben. Bitte über die Übersicht ein Rezept auswählen.";
} else {
Ohne ID in der URL wird gar nicht erst versucht, etwas zu laden — direkt die Fehlermeldung.
  ladeRezeptNachId(rezeptId)
    .then(rezept => {
      aktuellesRezept = rezept;
      if (!aktuellesRezept) {
        statusElement.textContent = "Kein Rezept mit dieser ID gefunden.";
        return;
      }
      aktuellePortionen = aktuellesRezept.portionen;
      rezeptAnzeigen(aktuellesRezept);
      inhaltElement.classList.remove("d-none");
    });
}
rezept ist entweder das gefundene Objekt oder null (siehe api.js) — if (!aktuellesRezept) fängt beide „nichts gefunden“-Fälle gleich ab. classList.remove("d-none") blendet den vorbereiteten, bis dahin versteckten Inhalt ein.
9 · Herzstück der Detailseite

Portionsrechner: Faktor statt Tabelle

Es wird keine Mengen-Tabelle für jede Portionszahl gespeichert. Stattdessen: ein Faktor, mit dem jede Zutatenmenge live neu berechnet wird.

Faktor = gewählte Portionen ÷ Grundportionen = 4 ÷ 2 = 2 Neue Menge = alte Menge × Faktor → 200 g Mehl × 2 = 400 g
js/rezept.js
function zutatenAnzeigen() {
  const faktor = aktuellePortionen / aktuellesRezept.portionen;
  const liste = document.getElementById("zutaten-liste");
  liste.innerHTML = "";
Reine Division zweier Zahlen — der ganze „Portionsrechner“ ist eigentlich nur diese eine Zeile.
  aktuellesRezept.zutaten.forEach(zutat => {
    const li = document.createElement("li");
    li.className = "list-group-item";
    const neueMenge = rundeMenge(zutat.menge * faktor);
    li.innerHTML = `<span>${zutat.name}</span><span>${neueMenge} ${zutat.einheit}</span>`;
    liste.appendChild(li);
  });
Für jede Zutat wird ein neues <li> gebaut, die Menge mit dem Faktor multipliziert und direkt eingehängt — deshalb erscheint die neue Menge sofort, ganz ohne Neuladen.
  document.getElementById("portionen-anzahl").textContent = aktuellePortionen;
}
textContent ändert nur den Text eines Elements, ohne die ganze Seite neu zu laden.
document.getElementById("portionen-plus").addEventListener("click", () => {
  aktuellePortionen++;
  zutatenAnzeigen(); // kein Neuladen — nur die Liste wird neu gebaut
});
() => {…} ist eine Arrow-Funktion ohne Parameter (leere Klammern) — läuft bei jedem Klick auf den Plus-Button erneut.

Genau das erfüllt zwei Kriterien auf einmal: „Portionsanpassung mit Faktorberechnung“ und „Zutatenanzeige aktualisiert sich ohne Neuladen“.

10 · Ein guter Lehrmoment

Nicht jede Zutat hat eine feste Menge

Echte Rezeptdaten sind unordentlich: „Salz, nach Geschmack“ hat keine sinnvolle Zahl. In recipes.json ist menge dann null, der Freitext steckt in hinweis. Ohne Abfangen würde zutat.menge * faktor zu NaN führen.

js/rezept.js
if (zutat.menge === null) {
  // Zutaten ohne feste Menge (z. B. "nach Geschmack") werden nicht skaliert
  li.innerHTML = `<span>${zutat.name}</span><span class="text-muted">${zutat.hinweis ?? ""}</span>`;
} else {
  const neueMenge = rundeMenge(zutat.menge * faktor);
  li.innerHTML = `<span>${zutat.name}</span><span>${neueMenge} ${zutat.einheit}</span>`;
}
=== null prüft exakt auf den Wert null (statt ==, um Verwechslungen mit 0 oder leerem Text zu vermeiden). ?? "" liefert einen leeren Text, falls sogar hinweis fehlt.

Von der Detailseite in den Browser-Speicher

Der Button „Zutaten zur Einkaufsliste hinzufügen“ schreibt die aktuell skalierten Mengen nach localStorage. Kommt dieselbe Zutat (gleicher Name + gleiche Einheit) schon vor, werden die Mengen addiert statt dupliziert:

Rezept A
200 g Mehl
+
Rezept B
150 g Mehl
→
Einkaufsliste
350 g Mehl
js/rezept.js
document.getElementById("btn-zur-einkaufsliste").addEventListener("click", () => {
  const faktor = aktuellePortionen / aktuellesRezept.portionen;
  const vorhandeneListe = JSON.parse(localStorage.getItem("einkaufsliste")) || [];
localStorage.getItem() liefert Text (oder null, wenn noch nichts gespeichert wurde). JSON.parse() macht daraus wieder ein JS-Array. || [] fängt den Fall „noch nichts gespeichert“ ab.
  aktuellesRezept.zutaten.forEach(zutat => {
    const neuerEintrag = {
      name: zutat.name,
      menge: zutat.menge !== null ? rundeMenge(zutat.menge * faktor) : null,
      einheit: zutat.einheit,
      rezeptTitel: aktuellesRezept.titel
    };
Für jede Zutat wird ein einfaches Objekt gebaut — genau die Werte, die später in der Einkaufsliste angezeigt werden. Der Ternär (? :) skaliert nur, wenn wirklich eine Menge vorhanden ist.
    // Gleiche Zutat aus dem gleichen Rezept wird zusammengeführt statt dupliziert
    const gleicheZutat = vorhandeneListe.find(eintrag =>
      eintrag.name === neuerEintrag.name && eintrag.einheit === neuerEintrag.einheit
    );
.find() gibt das erste passende Element zurück (oder undefined, wenn keins passt) — anders als .filter(), das immer ein ganzes Array liefert.
    if (gleicheZutat) {
      gleicheZutat.menge = rundeMenge(gleicheZutat.menge + neuerEintrag.menge);
    } else {
      vorhandeneListe.push(neuerEintrag);
    }
  });
.push() hängt ein neues Element ans Ende eines Arrays an — nur wenn kein passender Eintrag gefunden wurde.
  localStorage.setItem("einkaufsliste", JSON.stringify(vorhandeneListe));
});
JSON.stringify() macht aus dem JS-Array wieder Text — localStorage kann nämlich nur Text speichern, keine echten Objekte/Arrays.
12 · Einkaufsliste

Anzeigen, bearbeiten, löschen — ganz ohne fetch()

einkaufsliste.html braucht keine API: alle Daten kommen aus localStorage. Löschen funktioniert über den Index in der gespeicherten Liste.

js/einkaufsliste.js
function listeAnzeigen() {
  const eintraege = JSON.parse(localStorage.getItem("einkaufsliste")) || [];
  listeElement.innerHTML = "";

  if (eintraege.length === 0) {
    statusElement.textContent = "Die Einkaufsliste ist noch leer. …";
    return;
  }
Gleiches Muster wie beim Speichern: Text aus localStorage lesen, mit JSON.parse zurück in ein Array verwandeln, || [] für den leeren Fall.
  eintraege.forEach((eintrag, index) => {
    const li = document.createElement("li");
    li.innerHTML = `<span>${eintrag.name} – ${eintrag.menge} ${eintrag.einheit}</span>
      <button>Entfernen</button>`;
forEach((eintrag, index) => …) kann neben dem Element auch den Index (die Position im Array, ab 0 gezählt) mitgeben — genau der Index, der gleich zum gezielten Löschen gebraucht wird.
    li.querySelector("button").addEventListener("click", () => eintragEntfernen(index));
    listeElement.appendChild(li);
  });
}
() => eintragEntfernen(index) — kurze Arrow-Funktion ohne geschweifte Klammern. Wichtig: jeder Button „merkt“ sich seinen eigenen index (Closure) und weiß so, welchen Eintrag er löschen soll.
function eintragEntfernen(index) {
  const eintraege = JSON.parse(localStorage.getItem("einkaufsliste")) || [];
  eintraege.splice(index, 1);
  localStorage.setItem("einkaufsliste", JSON.stringify(eintraege));
  listeAnzeigen();
}
.splice(index, 1) entfernt genau 1 Element ab Position index — direkt aus dem Array, nicht nur aus der Anzeige. listeAnzeigen() danach baut die komplette Liste neu auf.
document.getElementById("btn-liste-leeren").addEventListener("click", () => {
  localStorage.removeItem("einkaufsliste");
  listeAnzeigen();
});
removeItem löscht den kompletten gespeicherten Wert auf einen Schlag — anders als splice, das nur ein einzelnes Element entfernt.

localStorage überlebt das Neuladen der Seite und sogar das Schließen des Browsers — genau das verlangt das Kriterium „Einkaufsliste speichern und laden“.
